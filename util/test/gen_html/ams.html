API APIs define sets of HTTP URL paths available on the World-wide Web

Historical Note: The APIProduct concept in Apigee Edge (circa 2016) is an amalgam of the API and APIProduct concepts

Includes properties and other constraints from PersistentResource
Property Name Property Type Property Description
kind string The value of this property indicates the 'kind' or 'type' of the object. Its value must be the string "API"
name string The name of the API. Must be unique amongst APIs of the same org. The name of the product, along with the name of the org, may be used to perform queries using URI templates of the form: https://api.enterprise.apigee.com/v1/organizations/{org-name}/apis/{api-name}
displayName The name to be used to display to humans for this API. Used in the UI. type: string
description string A textual description of the API
machine_readable_description string Defines the URI paths to which this API gives access. The format of the description is variable. The simplest is a fragment URI that points to an array of paths in the same document. An alternative is to point to an OpenAPI Specification.
environments [0:1 (Environment)] Environments where this API is available. Defines the domain names where the paths of the API will be available
ui_namesOfProxiesProvidingPaths [string] The API is specified by the uriPaths property, which defines the paths of the API. As a convenience, the UI allows the uriPaths to be specified by picking from the list of paths supported by a particular proxy. If the user does this, the UI will use this property to remember the names of proxies that were used as sources

Subscription Subscriptions record the usage of an Subscription by a subscriber.

Historical Note: The App concept in Apigee Edge (circa 2016) is similar to the Subscription concept, to the point where it can be considered a renaming. App was not the best choice of name because the concept being modelled is not an app, it is just a key-protected usage of an APIProduct. Using one of these subscriptions per app is only a suggestionâ€”it is not possible to know whether customers use a single subscription for a single app, a single subscription for many apps, or a many subscriptions for one app.

The Key/Secret and the products they apply to. Unlike The 2016 Edge "App" concept, only one key per Subscription is supported. Edge was motiviated to allow many because they thought they were modeling apps, and apps may use multiple keys. Since subscriptions are not apps, and apps are free to use as many subscriptions as they like, there is no need to support multiples keys per Subscription
